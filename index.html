What makes a summary meaningful?
Relevance to you. A summary that tells a Finance user about acquisition negotiations is noise. One that leads with budget variance is signal. Right record + right role + right timing.
What if Scout hasn't seen this object type before?
Compass provides the schema by default. The agent reads the Compass ontology — field types, picklist values, SOQL guidance, related objects — and reasons about what matters. If Compass doesn't know the object yet, universal queries (Tasks, Chatter, Notes, Emails, Events) work on every record.
How does Scout know what would be meaningful to YOU?
From the first call, your role and profile (from Salesforce) shapes the focus. Over time, usage patterns across everyone on this object type build shared context. Eventually, your own interaction history builds a personal micro-context — fed into every future summary.
How does Scout have enough context to produce the summary?
Four inputs before a single query runs: (1) Compass ontology for this object. (2) User name, role, profile. (3) Today's date — for staleness and overdue calculations. (4) The record ID to scope all queries. Then 7+ parallel data fetches. Then synthesis.
Architecture

What happens on a request. What happens in the background.

From the user's perspective, there is one action: Refresh Summary. What runs behind that button — and what runs invisibly between sessions — is what makes the summary get better over time.

⚡
On every request
Triggered by "Refresh Summary" or on a schedule
1
Locate. Read objectApiName, recordId, and user role from the Salesforce platform context.
2
Orient via Compass. Fetch the schema ontology for this object — fields, picklist values, SOQL guidance, related objects, and any learned context accumulated from prior use.
3
Fan out 7+ parallel queries. Record fields, Tasks, Chatter, Notes, Emails, Events, object-specific relations — all at once.
4
Synthesize. LLM reads all data plus any user-specific or object-specific context that has accumulated. Writes the structured summary and 3–5 suggested follow-up prompts.
5
Deliver inline. Summary renders in the record page. Prompts appear as action chips. User can continue with Scout for a full session.
🔄
In the background
Runs on a schedule — not visible to the user
Triggered by usage thresholds — runs more often as activity grows
1
Analyze execution history. A background agent queries traces filtered by object type — which fields appeared in the best summaries, which action chips users clicked.
2
Update the Compass ontology. Writes a tuneHeader to the object's Compass document: "For Site records, users consistently need ownership status and LOI timeline first." Every future summary inherits this automatically.
3
Build user context. After enough interactions per user, a personalization agent reads their trace and feedback history, writes a user_context block into Mission defaults — shaping the tone, order, and focus of their next summary.
4
No action needed. The next refresh is measurably better. The more the team uses it, the faster and more precisely it learns — an active org can reach peak intelligence within days, not months.

Compass Ontology layers — where STRK objects sit today
1
Layer 1 — Full knowledge
Account, Opportunity, Contact, Case — deepest ontology, semantic field understanding
2
Layer 2 — Surveyed
strk2__stf_* objects surveyed today. strk__Site__c, Engagement, Project need Magellan survey — this is the infrastructure task.
3
Layer 3 — Stub
Schema only — field names and types but no SOQL guidance or semantic context. Summaries fall back to universal queries.
Current gap: Core STRK objects (strk__Site__c, strk__Engagement__c, strk__Project__c) are at Layer 3. The Compass Survey infrastructure task elevates them to Layer 2 — unlocking field-rich, SOQL-guided summaries for these objects.

How It Learns

The same output. Growing intelligence.

The user never sees the learning happen. What they see is the summary getting progressively more useful — leading with the right things, surfacing the right risks, suggesting the right follow-ups. Here is what runs in the background to make that happen.

Maturity is driven by usage, not time
An active team can reach full intelligence within the first week. Time is just how long it takes to accumulate enough summaries.
Every summary
Always on, from the first use
Role-aware. Knows the schema, surfaces risks, suggests next steps. No threshold — works immediately and never turns off.
~50 team summaries
Days with an active team
Object patterns emerge. Compass learns which fields and signals matter most. Suggested prompts become record-specific, not generic.
~20 personal uses
A week for a regular user
Your context builds. The summary reorders itself around what you care about — role, habits, the sections you always re-read.
⚡ In a team that generates 10–15 summaries a day, the object-level intelligence typically kicks in within the first 3–5 days. A user who runs summaries daily reaches their personal context within 3–4 weeks. Both thresholds are configurable.

Object-level learning — shared by everyone

Once enough summaries of a given object type have been generated, a background agent analyzes the patterns and writes them into the Compass ontology. The more the team uses it, the sooner this kicks in — and the better it gets with every batch.

What to Build

Scope breakdown.

What is Salesforce-side, what is Scout-side, and what infrastructure needs to be set up first.

Salesforce (LWC + Apex)

⚡

c-st-scout-summary (LWC)

Embedded inline in the record page. Accepts @api recordId + @api objectApiName from the platform. Renders the summary, action chips, and follow-up entry point inline. Droppable on any flexipage via Lightning App Builder.

Build new
🔧

StScoutSummaryController (Apex)

@AuraEnabled controller. Uses Schema.describeSObjects for field context. HTTP callout to Scout API using the StAppConfigKey credential pattern (same as StSlackAPI.cls). Returns summary text and suggested prompts.

Build new
🔑

Sitetracker_API_Keys.SCOUT_API_KEY

Custom metadata record in the existing Sitetracker_API_Keys__mdt type. Identical pattern to SLACK_API_KEY already in the codebase.

Extend existing
📋

StScoutLearningScheduler (Apex Scheduler)

Apex job that triggers the background learning graphs once a usage threshold is crossed per object type — e.g. every 50 new summaries. Runs more frequently for high-activity objects. This is what makes the summary get smarter without anyone doing anything.

Build new

Scout Platform (Graphs + Compass)

🗂

Universal Summary Graph

New graph: context_builder → compass_query → parallel_fan_out (7 queries) → parallel_join → llm_call (summarize + prompts) → emit_artifact. Accepts record_id, object_api_name, user_role, and optional user_context. Produces the summary and suggested prompts.

New graph
🧠

Object Learning Graph

Background graph: receives object_api_name → queries recent traces → llm_call to extract patterns → writes a tuneHeader to the Compass ontology for that object → refreshes suggested prompts. Run on a schedule by the Apex scheduler.

New graph
👤

User Personalization Graph

Background graph: queries traces and feedback by userId → llm_call to build user_context string → stores in Mission inputDefaults. Invoked by the scheduler per user-object combination after enough interactions.

New graph
📡

Compass Survey — STRK Objects

Add a STRK-enabled org as a Compass connector policy. Run Magellan survey to elevate strk__Site__c, strk__Engagement__c, strk__Project__c from stub (layer 3) to fully surveyed (layer 2). Required for field-rich summaries on core STRK objects.

Open Questions

Decisions before we build.

A few things to decide before implementation starts. These affect scope and architecture — not feasibility.

1. Schedule or on-demand?

Does the summary generate automatically on a schedule (e.g. daily, so the user always sees a fresh summary on open) — or only when the user clicks Refresh? Auto-schedule feels more ambient but adds background compute per record.

On-demand only Daily scheduled refresh On-demand + staleness indicator

2. Where does it live in the UI?

The screenshot shows it embedded inline in the Engagements section. It could also live in a right-column sidebar component, or in a dedicated Summary tab. Each changes what the user's eye hits first.

Inline in Engagements tab Right column component Dedicated Summary tab

3. What does "Continue with Scout" open?

Should the CTA open a full Scout chat session (in a modal or new page) with this record pre-loaded as context? Or does an inline chat appear below the summary, staying within the record page?

Inline follow-up chat Opens Scout modal session Opens Scout in a new tab

4. Pilot object

Which object gets the first full experience — including Compass survey, object learning graph, and personalization? This determines what Magellan survey to prioritize and what training data to build first.
